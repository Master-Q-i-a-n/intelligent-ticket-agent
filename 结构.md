# 智能工单 Agent 系统结构说明

本文档用于快速理解当前项目结构。项目由三部分组成：

- `workOrderFrontend-main`：Vue 前端
- `workOrderBackend-main`：Spring Boot 业务后端
- `workOrderAI`：Python FastAPI AI 服务

## 1. 总体架构

```text
用户 / 管理员
    |
    v
Vue 前端
    |
    | HTTP / Axios
    v
Spring Boot 业务后端
    |
    | JDBC
    v
MySQL

Spring Boot 业务后端
    |
    | HTTP 转发 AI 请求
    v
Python FastAPI AI 服务
    |
    |  LangGraph / Tool Calling / RAG
    v
qwen 模型 + Milvus Lite 向量库
```

## 6. 核心业务流程

### 6.1 用户创建工单

```text
用户提交反馈
  -> Vue 调 Java /feedback 接口
  -> Java 写入 wo_feedback，初始客服组默认为产品咨询组
  -> Java 异步调用 Python /ai/classify
  -> Python 返回 category / priority / emotion
  -> Java 更新工单分类、优先级、情绪，并按分类重算客服组
```

客服组路由规则：

- 技术故障 -> 技术支持组
- 账单问题 -> 财务客服组
- 产品咨询 / 功能需求 / 投诉建议 / 其他 / UNKNOWN -> 产品咨询组

### 6.2 管理员生成 AI 回复建议

```text
管理员点击 AI 建议
  -> Vue 调 Java /work-order/{id}/reply-suggestion
  -> Java 查询工单详情和完整回复历史
  -> Java 请求 Python /ai/suggestion，并带上 owner_username
  -> Python SuggestService 调用 ReplySuggestionGraph
  -> LangGraph 固定执行：
       1. 整理工单上下文
       2. 前置检索相似历史案例
       3. qwen3-32b 语义路由
       4. 按路由进入知识问答、故障诊断、用户记录、澄清或拒答分支
       5. 生成客服回复草稿
       6. self-check 检查事实依据、权限隐私和完整性
       7. 必要时重写 1 次，仍失败则安全降级
  -> USER_RECORD 分支保留局部 ReAct，只允许调用：
       - get_time_now
       - fetch_current_user_records
       - rag_summarize
  -> Python 返回 suggestedReply 和历史案例来源
  -> Java 转发给前端
  -> Vue 填入富文本回复框，并展示参考历史案例
```

### 6.3 知识问答

```text
用户 / 管理员提问
  -> Vue 调 Java /knowledge/qa
  -> Java 转发 Python /ai/knowledge/qa
  -> Python RagService：
       1. HyDE 前置分类
       2. 必要时生成短假设性检索文本
       3. 混合检索召回候选
       4. rerank 重排
       5. 生成答案和来源
  -> 前端展示答案与引用来源
```

### 6.4 知识库上传与删除

```text
管理员上传 pdf / docx / txt
  -> Java 做管理员权限校验
  -> Java 转发 Python AI 服务
  -> Python 保存文件元数据到 wo_knowledge_document
  -> Python 切分文档并写入 Milvus Lite rag_collection

管理员删除文档
  -> Python 删除元数据
  -> Python 按 document_id 删除对应向量分块
  -> 失效检索缓存
```

### 6.5 历史案例记忆

```text
工单状态变为 SOLVED / CLOSED
  -> Java 获取最后一条客服回复
  -> Java 调 Python /ai/case-memory
  -> Python 写入 wo_case_memory
  -> Python 将问题 + 最终回复写入 case_memory_collection

后续生成回复建议
  -> SuggestService 先检索相似案例
  -> 命中后注入 Agent 上下文
  -> 返回 sourceTemplates 给前端展示
```

## 7. RAG 当前实现

```text
知识文档
  -> 文件读取
  -> 文本切分
  -> Embedding
  -> Milvus Lite rag_collection

查询时
  -> HyDE 判断查询类型
  -> Question / Fault Statement 生成短检索扩展文本
  -> 向量检索 + BM25 混合召回
  -> rerank_candidate_k = 5
  -> qwen3-rerank 重排
  -> rerank_top_k = 3
  -> 总结模型生成答案
```

回复建议里的轻量 RAG 不走完整 HyDE，直接检索并 rerank，用于降低生成建议耗时。RAG 工具内部保留结构化 `summary + sources`，前端回复建议区域默认只展示历史案例来源，不展示 RAG 来源。

## 8. 评测体系

```text
workOrderAI/evals/
├─ datasets/
│  ├─ classification.jsonl
│  ├─ reply_suggestion.jsonl
│  └─ knowledge_qa.jsonl
├─ runner.py              # 执行三类任务
```

三类评测任务：

- `classification`：问题类型、优先级、情绪
- `reply_suggestion`：工具调用、事实依据、证据链、LLM Judge
- `knowledge_qa`：来源命中、拒答、正文规则、LLM Judge

## 9. 当前项目边界

这是学习 / 面试展示项目，不是生产级系统。当前仍保留以下演示性质设计：

- 密码明文存储
- Java session 存在内存中
- 公开注册可选择管理员角色
- Java 业务服务与 Python AI 服务之间没有服务鉴权
- Milvus Lite 适合本地演示，不适合多人并发生产环境

如果要生产化，需要补充密码加密、服务间鉴权、管理员审批、权限隔离、审计日志、限流、告警、Redis session 或 JWT，以及正式向量数据库部署。
